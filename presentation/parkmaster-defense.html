02 · System design

Security model

Authentication
  • Passwords stored as BCrypt hashes - plaintext never persisted or logged
  • Login issues a stateless JWT carrying the role claim
  • JwtAuthFilter validates every request
  • Forgot password: single-use reset token
Authorization - two layers
  • Coarse: RBAC by URL prefix in SecurityConfig - /api/admin/**, /manager/**, /staff/**, /driver/**
  • Fine: object ownership in the service layer - e.g. getForUser(email, id) so a driver only reads their own session
  • Public endpoints isolated under /api/public/**

03 · Workflows

Driver: reserve → park → pay

Free tier
  • AI assigns the slot automatically
  • 10% checkout discount for free-reservation sessions
  • Hold expires 30 min after reserved start (hold_until)
  • QR code shown to staff at the gate
Paid tier
  • Driver picks the exact slot
  • VNPay deposit = 1-hour rate
  • Slot locked RESERVED on payment
  • Guaranteed on arrival
While parked
  • Live status on My Parking
  • Session history + feedback
  • Monthly pass → checkout charge = 0
  • Pay at exit via VNPay or cash

03 · Workflows

Staff: check-in / check-out

Check-in
  • Scan QR - RES:<id> or PASS:<plate> - or search plate/ticket
  • Walk-in: AI Auto (score breakdown shown) or Manual slot pick
  • Slot flips AVAILABLE → OCCUPIED, session created with ticket code
Check-out
  • Find session on Active Sessions (live slot map)
  • Charge computed, panel shows amount due
  • Settle cash on-site, or driver pays VNPay on their phone
  • Active monthly pass → charge 0, session completes instantly
charge = ceil_hours(minutes − grace) × rate_per_hour, capped at daily_cap per started day, × peak_multiplier

03 · Workflows

Payments - VNPay integration

Outbound - build checkout URL
  • Sort params, URL-encode name=value pairs
  • Sign with HMAC-SHA512 + merchant secret
  • Unique gatewayRef (txnRef) persisted before redirect
Return - verify & complete
  • Recompute signature over all vnp_* params
  • Match payment by gatewayRef, check code 00 + amount
  • Set PAID idempotently, then complete the linked session / pass / reservation
Methods: CASH (staff settle) · ONLINE (mock settle) · VNPAY (sandbox gateway) - one payment row per session, status PENDING → PAID
Speaker Trọng
11 / 18

Why idempotent: VNPay may redirect + retry - double PAID must not double-complete. Amount check prevents tampering with return params. ~2 min.

04 · AI allocation

Scoring - max 100 points

40 · Vehicle match

Binary: slot's floor dedicated to this vehicle type → 40; mixed floor with no dedicated type → 20; wrong type → 0.

30 · Load balance

Floor's availableRatio × 30 - spreads vehicles so no floor saturates while others sit empty.

20 · Distance

20 / floor.level - ground floor scores full 20, each level up scores less. Proxy for time-to-park.

10 · Peak hour

Bonus only when PeakHours.isPeakNow() - 7–9 AM and 5–7 PM, Asia/Ho_Chi_Minh.

rank() loads all AVAILABLE slots in the building, scores each, returns best-first · allocate() = top-1 or HTTP 409 · score persisted as JSONB per session
Speaker Vũ
13 / 18

Weights defense: vehicle match dominates because a wrong-type slot is useless; load balance beats distance because building throughput > individual walk. Weights are constants - tunable, and persisted scores let us re-evaluate them (RQ3). ~2.5 min.

04 · AI allocation

Research questions

RQ1 · Segmentation

Does floor/zone segmentation by vehicle type affect utilization? → floor.vehicle_type + per-floor fill-rate analytics

RQ2 · Auto vs free choice

Does auto allocation reduce time-to-park? → allocation-comparison report: auto vs manual session counts + avg duration

RQ3 · Which criteria matter

Distance, floor, type, time, fill rate - which weighs most? → per-session allocation_score JSONB keeps every component queryable

RQ4 · Peak utilization

Can the algorithm improve peak-hour utilization? → peak bonus in scoring + check-ins-by-hour distribution
